# Repository Guidelines

## Project

**mdroid** is a Factory Droid plugin that packages a production-grade engineering workflow. It combines OpenCodeKit artifacts lifecycle, agent-skills anti-rationalization patterns, and Superpowers TDD discipline, optimized for Droid-native features (Task tool, hooks, MCP, Missions). Its standout feature is a 5-layer automated memory system built with Droid hooks + Python that learns across sessions without spending LLM tokens.

## Project Structure & Module Organization

```
mdroid/
├── .factory-plugin/        # Plugin manifest (plugin.json) + marketplace.json
├── droids/                  # 8 custom subagents (explore, plan, review, oracle, scout, vision, security-audit, general)
├── skills/                  # 33 skills: workflow commands (/spec, /build, /verify, /ship...) + background skills + domain/design vocabulary
│   └── <name>/SKILL.md      # Each skill lives in its own dir as SKILL.md (some with disclosed reference .md files)
├── hooks/                   # 5-layer memory system: 7 hook scripts + lib/ (Python: tfidf, bm25, patterns, markdown, tokenize)
│   └── lib/                 # Shared Python modules used by hook scripts
├── references/              # 8 reference checklists (definition-of-done, testing, security, performance, code-smells, etc.)
├── templates/               # Artifact templates (spec-lite, spec-full, plan, progress)
├── mcp.json                 # MCP server configs (empty by default; users add their own)
├── README.md                # Plugin documentation
└── BUILD_PLAN.md            # Master build plan and decision log
```

Runtime state (generated by hooks, gitignored): `.factory/memory/`, `.factory/memories.md`, `.factory/artifacts/`, `.factory/tech-stack.md`. The domain glossary `CONTEXT.md` and `docs/adr/` live at the repo root of the *project using mdroid* (not the plugin repo itself).

## Build, Test, and Development Commands

This is a plugin package, not a buildable application. There is no compile step.

- **Install locally for testing:** `droid plugin marketplace add ~/workspace/mdroid && droid plugin install mdroid@mdroid`
- **Install from GitHub:** `droid plugin marketplace add https://github.com/minhduydev/mdroid && droid plugin install mdroid@mdroid`
- **Test hook scripts:** `echo '<json>' | python3 hooks/track-artifacts.py` (pipe a JSON stdin payload; verify output files under `.factory/memory/`)
- **Validate plugin manifest:** confirm `.factory-plugin/plugin.json` parses and `marketplace.json` lists the plugin

When bootstrapping a project that *uses* mdroid, run `/init` to generate `AGENTS.md`, `.factory/tech-stack.md`, `.factory/rules/` stubs, and a `CONTEXT.md` glossary stub, then `/spec` to start a feature.

## Coding Style & Naming Conventions

- **Droid files** (`droids/*.md`): YAML frontmatter with `name`, `description`, `model` (can override to a stronger model, e.g. `oracle` uses a high-reasoning model), and `tools: ["..."]` array. Kebab-case filename matching the droid name.
- **Skill files** (`skills/<name>/SKILL.md`): YAML frontmatter with `name`, `description` (must include a "Use when..." trigger), and flags like `user-invocable: true` / `disable-model-invocation: true`. Skill dir name is kebab-case.
- **Hook scripts** (`hooks/*.py`, `hooks/*.sh`): `#!/usr/bin/env python3` shebang, executable bit set. Python stdlib only (no external dependencies). Single script per hook event concern.
- **Shared lib** (`hooks/lib/*.py`): pure functions, importable by hook scripts via `from lib import ...`.
- **Line limits** (enforced by `hooks/structural-check.sh`): `SKILL.md` files <300 lines, `droids/*.md` files <500 lines, `.factory/rules/*.md` <200 lines, kebab-case filenames only, no `.env` files.
- **General**: Markdown for all docs and skills. Keep commands lean; prefer concise tables over verbose prose.

## Testing Guidelines

There is no unit-test framework for the plugin itself. Verification is manual and behavior-driven:

- **Hook smoke test:** pipe a representative JSON payload into each hook script and confirm the expected file is written under `.factory/` (e.g., `session-state.json`, `distillations/<id>.md`, `memories.md`).
- **Memory loop test:** run `track-artifacts.py` -> `save-summary.py` -> `distill-and-curate.py` -> `inject-memory.py` in order and confirm the learning loop produces injectable context.
- **Command test:** run `/init`, then `/spec "test feature"`, `/build`, `/verify`, `/review`, `/ship` end-to-end in a scratch project and confirm artifacts appear under `.factory/artifacts/<slug>/`.

## Commit & Pull Request Guidelines

- **Conventional commits** (observed in history): `feat:`, `fix:`, `refactor:` prefixes. Subject line lowercase, imperative mood.
- **Body** (when needed): blank line after subject, then bullet points explaining the why and what changed.
- **Co-authorship:** commits made via Droid include `Co-authored-by: factory-droid[bot] <138933559+factory-droid[bot]@users.noreply.github.com>`.
- **Staging:** stage specific files only. Never `git add .` (the main agent's commit protocol forbids it).
- **Per-task commits:** when implementing via `/build`, commit after each TDD task with a type prefix (`feat`, `fix`, `test`, `refactor`).
- **Pre-commit checks:** `hooks/structural-check.sh` runs on every Create/Edit/ApplyPatch and blocks commits that violate line limits or filename conventions.

## Agent-Specific Instructions

- The main interactive Droid owns implementation directly via `/build`. Subagents (`explore`, `review`, `scout`, `plan`, `vision`, `security-audit`, `general`) are for isolation or specialized capabilities, not for duplicating the main agent's edit ability.
- Read-only droids (`explore`, `review`, `vision`, `security-audit`) must never modify files; report findings with file:line citations instead.
- Every command skill includes an **anti-rationalization table** and a **Red Flags** section. Honor them: "seems right" is never sufficient evidence.
- Use the Task tool (not workflow files) for multi-step delegation. Use AskUser (not inline questions) for clarifications.
- Memory is automatic. Prefix a message with `#` to save a project memory, or `##` for a personal memory. Never edit `.factory/memories.md` by hand unless fixing corruption.

## Domain Language & Design Vocabulary

- **`CONTEXT.md`** (at the repo root of the project using mdroid) is the domain glossary. It is maintained inline by the `domain-modeling` skill as terms resolve during `/spec` and `/build`. It is a glossary only — no implementation details, file paths, or code. The `domain-modeling` skill sharpens fuzzy terms, challenges overloaded words, and offers ADRs sparingly (only when hard-to-reverse + surprising + real-trade-off).
- **ADRs** live in `docs/adr/` of the project using mdroid, numbered `0001-slug.md`. Create the directory lazily.
- **`codebase-design`** is the shared deep-module vocabulary (module, interface, depth, seam, adapter, leverage, locality). `/build` agrees seams up front using it; `/improve-codebase-architecture` surveys for deepening opportunities using it.
- Use `CONTEXT.md` vocabulary for the domain and `codebase-design` vocabulary for the architecture in every suggestion, review, and plan. Don't drift into "component," "service," "API," or "boundary."

## Skill Craftsmanship

- Skills are written following the `writing-great-skills` discipline: **leading words** recruit priors the model already holds (e.g. *tight* loop, *deep module*, *seam*, *red*, *tracer bullet*) — repeat the token, never the sentence. **Completion criteria** are checkable and exhaustive. **Progressive disclosure** pushes reference behind context pointers (sibling `.md` files) so the SKILL.md top stays legible. Each meaning has a **single source of truth**; hunt **no-ops** (lines the model already obeys by default) and delete them.
- The **no-op test is model-relative**: mdroid targets `model: inherit` and may run models other than Claude. Verify each leading word against the model actually in use before committing to it.
- Disclosed reference files (`GLOSSARY.md`, `CONTEXT-FORMAT.md`, `ADR-FORMAT.md`, sibling `.md` files) are part of the skill and loaded only when their context pointer fires.

## Router Maintenance Rule

The `development-lifecycle` skill is the router over every user-reachable command. Whenever a skill is added, renamed, removed, or its behaviour changes, re-read `development-lifecycle`'s `SKILL.md` and update the map so it stays accurate — a new command it never mentions, or a stale one it still routes to, is a router that lies. The same trigger applies to the top-level `README.md` command table: every command must have an entry.
